# ADR-010: DO-First Architecture with Command Handlers as Methods

**Status:** Proposed
**Date:** 2025-11-23
**Context:** Architectural refactoring to eliminate stateless mode and move command handlers from separate classes to methods on aggregates

---

## Decision

Move from **stateless CommandRoute pattern** to **DO-first pattern** where command handlers are methods on aggregate classes.

### Current Architecture (Stateless Mode)

```typescript
// Separate command handler class
@CommandHandler({
  route: '/accounts/:id/withdraw',
  params: WithdrawParams,
  body: WithdrawBody
})
export class WithdrawHandler extends UpdateCommandRoute<
  WithdrawCommand,
  AccountState,
  MoneyWithdrawnEvent
> {
  protected getStateClass() { return AccountState; }

  async executeUpdate(command: WithdrawCommand, state: AccountState) {
    if (state.balance < command.amount) {
      throw new BusinessRuleViolationError('Insufficient funds');
    }
    return new MoneyWithdrawnEvent(command.amount);
  }
}

// Application orchestrator
const app = new CevesApp(env);
app.fetch(request); // → Hono → CommandRoute.handle() → Restore state → Execute → Persist event
```

### New Architecture (DO-First)

```typescript
// Command handler as method on aggregate
export class BankAccountAggregate extends AggregateObject<AccountState> {
  constructor(ctx: DurableObjectState, env: any) {
    super(ctx, env, AccountState);
  }

  @CommandHandler({
    commandType: 'Withdraw',
    createCommand: false,  // Requires existing DO
    route: '/accounts/:id/withdraw',
    method: 'POST',
    params: WithdrawParams,
    body: WithdrawBody
  })
  async withdraw(command: WithdrawCommand): Promise<MoneyWithdrawnEvent> {
    // Can access this.state directly!
    if (this.state.balance < command.amount) {
      throw new BusinessRuleViolationError('Insufficient funds');
    }
    return new MoneyWithdrawnEvent(command.amount);
  }
}

// Worker routing layer (thin)
export default {
  fetch(request, env) {
    const match = routeToHandler(request);
    const stub = env.BANK_ACCOUNT.get(id);
    return stub.fetch(request); // → DO.fetch() → Route to method → Execute → Persist
  }
};
```

### ⚠️ **CHANGE** - Significant Modifications Required

#### 1. AggregateObject.ts - Add Command Router

**New Responsibilities:**
- Implement `fetch()` router that uses handler registry
- Restore state BEFORE calling decorated method
- Persist event AFTER method returns
- Handle errors and return HTTP responses

```typescript
class AggregateObject<TState> extends DurableObject {
  async fetch(request: Request): Promise<Response> {
    // 1. Parse request to get commandType
    const { commandType, commandData } = await this.parseRequest(request);

    // 2. Find handler method via registry
    const handler = getCommandHandler(this.aggregateType, commandType);

    // 3. Restore state BEFORE calling method
    await this.restoreState();

    // 4. Call the decorated method (user's business logic)
    const event = await this[handler.methodName](commandData);

    // 5. Persist event and update state
    await this.applyEvent(event);

    // 6. Return HTTP response
    return Response.json({ success: true, version: this.state.version });
  }
}
```

#### 2. @CommandHandler Decorator - Complete Rewrite

**Current:** Decorates classes, stores class constructors
**New:** Decorates methods, stores method names + metadata

```typescript
// NEW signature
interface CommandHandlerMetadata {
  commandType: string;
  methodName: string;          // NEW: Method name on aggregate
  aggregateType: string;        // Derived from class
  createCommand: boolean;       // NEW: Create vs update

  // HTTP/OpenAPI metadata (for worker layer)
  route: string;
  method: 'GET' | 'POST' | 'PUT' | 'DELETE';
  params?: z.ZodSchema;
  body?: z.ZodSchema;
  responses?: Record<number, z.ZodSchema>;

  doBinding: string;            // e.g., 'BANK_ACCOUNT'
}

export function CommandHandler(metadata: Partial<CommandHandlerMetadata>) {
  return function (
    target: any,                // Aggregate instance
    methodName: string,         // Method name
    descriptor: PropertyDescriptor
  ) {
    const aggregateType = target.constructor.name;
    const key = `${aggregateType}:${metadata.commandType}`;

    COMMAND_HANDLERS.set(key, {
      ...metadata,
      methodName,               // Store for routing
      aggregateType,
      doBinding: aggregateTypeToBinding(aggregateType)
    });
  };
}
```

#### 3. Worker Routing Layer - New Component

**Responsibilities:**
- Match HTTP routes to command handlers
- Validate requests against schemas
- Get DO stub by aggregateId
- Forward request to DO

```typescript
// src/core/WorkerRouter.ts (NEW FILE)
export function createWorkerRouter(env: Env) {
  return {
    async fetch(request: Request): Promise<Response> {
      // 1. Match route to handler
      const match = matchRouteToHandler(request.url, request.method);
      if (!match) return new Response('Not found', { status: 404 });

      // 2. Validate request
      const validated = await validateRequest(request, match.handler);

      // 3. Get DO stub
      const doNamespace = env[match.handler.doBinding];
      const id = doNamespace.idFromName(validated.params.id);
      const stub = doNamespace.get(id);

      // 4. Forward to DO
      return stub.fetch(request);
    }
  };
}
```

## Migration Impact

### Breaking Changes

1. **Command handlers are now methods, not classes**
   - Users must refactor handler classes → methods on aggregates
   - Pattern: 1 aggregate class with N command handler methods

2. **No more CevesApp**
   - Users must use new Worker router
   - Pattern: `createWorkerRouter(env).fetch(request)`

3. **CreateCommandRoute / UpdateCommandRoute removed**
   - Replaced by `createCommand: boolean` flag in decorator

4. **getStateClass() method removed**
   - AggregateObject already has StateClass from constructor

### Migration Path

#### Step 1: Move command handlers to aggregate

```typescript
// BEFORE: Separate files
// example/src/commands/OpenAccountHandler.ts
@CommandHandler({ route: '/accounts/:id/open' })
export class OpenAccountHandler extends CreateCommandRoute { }

// AFTER: Methods on aggregate
// example/src/aggregates/BankAccountAggregate.ts
export class BankAccountAggregate extends AggregateObject<AccountState> {
  @CommandHandler({ commandType: 'OpenAccount', createCommand: true })
  async openAccount(command: OpenAccountCommand) { }
}
```

#### Step 2: Update worker entry point

```typescript
// BEFORE: CevesApp
import { CevesApp } from 'ceves';
export default {
  fetch(request, env) {
    const app = new CevesApp(env);
    return app.fetch(request);
  }
};

// AFTER: Worker router
import { createWorkerRouter } from 'ceves';
export default createWorkerRouter();
```

#### Step 3: Update tests

```typescript
// BEFORE: Test handler class
const handler = new OpenAccountHandler(eventStore, snapshotStore);
await handler.handle(request);

// AFTER: Test aggregate method
const aggregate = new BankAccountAggregate(ctx, env);
const event = await aggregate.openAccount(command);
```

## Benefits

### 1. Simplified Architecture
- ✅ **One pattern** instead of two (stateless vs DO mode)
- ✅ **Fewer files** - handlers co-located with aggregate
- ✅ **No duplication** - getStateClass() eliminated
- ✅ **Clear ownership** - aggregate owns its commands

### 2. Better Developer Experience
- ✅ **Direct state access** - `this.state` instead of parameter
- ✅ **Type safety** - TypeScript knows this.state type
- ✅ **Simpler testing** - Just call methods
- ✅ **Co-location** - All aggregate behavior in one file

### 3. Correctness
- ✅ **DO invocation model** - Aligns with how DOs work
- ✅ **Clean architecture** - Domain methods can be separate from command handlers
- ✅ **Less boilerplate** - No need for separate base classes

---

## Drawbacks

### 1. Migration Cost
- ⚠️ **Breaking change** - All existing projects must migrate
- ⚠️ **Documentation** - Need migration guide
- ⚠️ **Examples** - All examples must be rewritten

### 2. File Organization
- ⚠️ **Larger files** - Aggregates with many commands could be large
- ⚠️ **Potential solution** - Can still split into separate files and import
