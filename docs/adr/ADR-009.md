# ADR-009: Class-Based State with Empty State Pattern

**Status:** Proposed
**Date:** 2025-11-22
**Context:** Epic 9 follow-up - Eliminate null checks in event handlers
**Related:** ADR-008 (Domain Event Pattern)

## Problem Statement

Current event handlers have several pain points:

1. **Null checks everywhere**: Every event handler manually validates `state !== null`
2. **Infrastructure field management**: Handlers manually set `version`, `orgId`, `timestamp`
3. **Type safety gaps**: `state: AccountState | null` requires runtime checks
4. **Boilerplate**: Same validation code repeated in every handler

```typescript
// Current pattern - lots of boilerplate
apply(state: AccountState | null, event: MoneyDepositedEvent, metadata: EventMetadata) {
  if (state === null) {  // Manual check
    throw new Error('MoneyDeposited event requires existing account state');
  }

  return {
    ...state,
    balance: state.balance + event.amount,
    timestamp: metadata.timestamp,  // Manual infrastructure field
    // version and orgId handled elsewhere, but inconsistent
  };
}
```

## Decision

**Convert state from interfaces to classes with an empty state factory pattern.**

### Key Changes

1. **BaseState abstract class** with common fields
2. **Domain states extend BaseState** with business fields
3. **Event handlers ALWAYS receive non-null state** (empty for first event)
4. **Framework auto-manages ONLY timestamp and version** (orgId is business logic)

### Architecture

```typescript
// 1. BaseState abstract class with common fields
abstract class BaseState {
  id: string = '';          // Aggregate identity (set by handlers)
  version: number = 0;      // Infrastructure - auto-set by framework
  orgId: string = '';       // Business field - set by handlers, read by framework
  timestamp: string = '';   // Infrastructure - auto-set by framework

  // Factory method for creating empty initial state
  static empty<T extends BaseState>(this: new () => T): T {
    return new this();
  }
}

// 2. Domain state extends BaseState with business fields only
class AccountState extends BaseState {
  owner: string = '';
  balance: number = 0;

  // Can override empty() for custom initialization if needed
  static empty(): AccountState {
    return new AccountState();
  }
}

// 3. Event handlers NEVER receive null - handlers set id and orgId
@EventHandler({ eventType: 'AccountOpened', aggregateType: 'account' })
class AccountOpenedHandler implements IEventHandler<AccountState, AccountOpenedEvent> {
  apply(
    state: AccountState,  // ALWAYS non-null! Empty for first event.
    event: AccountOpenedEvent,
    metadata: EventMetadata
  ): AccountState {
    // No null check needed!
    // For first event: state === AccountState.empty()
    // For subsequent events: state has actual data

    return {
      ...state,
      id: metadata.aggregateId,  // Handler sets id (from metadata)
      orgId: metadata.orgId,     // Handler sets orgId (business decision)
      owner: event.owner,
      balance: event.initialDeposit
      // timestamp and version are auto-set by framework AFTER this returns
    };
  }
}

@EventHandler({ eventType: 'MoneyDeposited', aggregateType: 'account' })
class MoneyDepositedHandler implements IEventHandler<AccountState, MoneyDepositedEvent> {
  apply(
    state: AccountState,  // ALWAYS non-null!
    event: MoneyDepositedEvent,
    metadata: EventMetadata
  ): AccountState {
    // No null check! Just update business logic
    return {
      ...state,
      balance: state.balance + event.amount
      // Framework updates timestamp automatically
    };
  }
}

// 4. Framework provides empty state and auto-manages timestamp/version only
function applyEventToState<TState extends BaseState>(
  aggregateType: string,
  state: TState | null,
  event: StoredEvent,
  StateClass: new () => TState
): TState {
  const handler = getEventHandler(event.type);

  // Provide empty state for first event, actual state for subsequent
  const inputState = state ?? StateClass.empty();

  // Handler applies business logic (sets id, orgId, and business fields)
  const newState = handler.apply(inputState, event.event, {
    aggregateId: event.aggregateId,
    version: event.version,
    timestamp: event.timestamp,
    orgId: event.orgId
  });

  // Framework ONLY sets timestamp and version (infrastructure fields)
  // Handler is responsible for setting id and orgId (business fields)
  newState.timestamp = new Date().toISOString();
  newState.version = event.version;

  return newState;
}

// When saving events, framework READS orgId FROM state
function createStoredEvent<TEvent extends DomainEvent>(
  state: BaseState,
  domainEvent: TEvent
): StoredEvent<TEvent> {
  return {
    aggregateType: state.constructor.name,
    aggregateId: state.id,
    version: state.version,
    orgId: state.orgId,        // READ from state (set by event handler)
    timestamp: new Date().toISOString(),
    type: domainEvent.type,
    event: domainEvent
  };
}

// 5. Aggregate classes know their state type
class AccountAggregate extends AggregateObject<AccountState> {
  // Generic parameter tells framework to use AccountState.empty()
  // for first event
}
```

## Benefits

### 1. Zero Null Checks in Handlers
```typescript
// Before: Manual null check in EVERY handler
if (state === null) throw new Error('...');

// After: No null checks needed - state is ALWAYS AccountState
apply(state: AccountState, event, metadata) { ... }
```

### 2. Type Safety
```typescript
// Before: state might be null
const balance = state?.balance ?? 0;  // Defensive

// After: TypeScript knows state is never null
const balance = state.balance;  // Direct access, type-safe
```

### 3. Framework Manages Infrastructure
```typescript
// Before: Handlers manually manage timestamp
return { ...state, timestamp: metadata.timestamp };

// After: Framework updates timestamp automatically
return { ...state, balance: newBalance };  // Just business logic!
```

### 4. Consistent Pattern
- First event: receives `AccountState.empty()`
- Subsequent events: receive actual state
- All handlers use same pattern - no special cases

### 5. Better Defaults
```typescript
class AccountState extends BaseState {
  owner = '';      // Default empty string
  balance = 0;     // Default zero

  // Can customize empty state if needed
  static empty(): AccountState {
    const state = new AccountState();
    state.balance = 0;  // Explicit zero balance for new accounts
    return state;
  }
}
```

## Implementation Details

### File Changes

**1. Update BaseState (src/schemas/State.ts)**
```typescript
// Change from interface to abstract class
export abstract class BaseState {
  id: string = '';
  version: number = 0;
  orgId: string = '';
  timestamp: string = '';

  static empty<T extends BaseState>(this: new () => T): T {
    return new this();
  }
}
```

**2. Update IEventHandler interface**
```typescript
// Remove | null from state parameter
export interface IEventHandler<TState extends BaseState, TEvent extends DomainEvent> {
  apply(
    state: TState,  // Changed from: TState | null
    event: TEvent,
    metadata: EventMetadata
  ): TState;  // Changed from: Omit<TState, 'version' | 'orgId'>
}
```

**3. Update applyEventToState utility**
```typescript
export function applyEventToState<TState extends BaseState>(
  aggregateType: string,
  state: TState | null,
  event: StoredEvent,
  StateClass: new () => TState
): TState {
  const handler = getEventHandler(event.type);

  // Provide empty state for first event
  const inputState = state ?? StateClass.empty();

  const metadata: EventMetadata = {
    aggregateId: event.aggregateId,
    version: event.version,
    timestamp: event.timestamp,
    orgId: event.orgId
  };

  // Apply business logic
  const newState = handler.apply(inputState, event.event, metadata);

  // Framework sets infrastructure fields
  newState.version = event.version;
  newState.orgId = event.orgId;
  newState.timestamp = new Date().toISOString();

  return newState;
}
```

**4. Update AggregateObject to pass StateClass**
```typescript
export abstract class AggregateObject<TState extends BaseState> {
  // Define abstract method for state class
  protected abstract getStateClass(): new () => TState;

  protected async applyEvent(event: StoredEvent): Promise<void> {
    // Pass state class to utility
    this.state = applyEventToState(
      this.aggregateType,
      this.state,
      event,
      this.getStateClass()
    );
  }
}
```

**5. Update example states to classes**
```typescript
// example/src/types.ts
export class AccountState extends BaseState {
  owner: string = '';
  balance: number = 0;
}

// Aggregate implementation
export class AccountAggregate extends AggregateObject<AccountState> {
  protected getStateClass() {
    return AccountState;
  }
}
```

### Migration Path

**Phase 1: Core Changes**
- Convert BaseState interface → abstract class
- Update IEventHandler to remove `| null` and `Omit<>`
- Update applyEventToState to use empty state pattern
- Update AggregateObject to support StateClass

**Phase 2: Example Migration**
- Convert AccountState to class
- Remove null checks from all event handlers
- Remove manual timestamp management
- Update tests

**Phase 3: Documentation**
- Update README migration guide
- Add empty state pattern examples
- Document class-based state approach

## Alternatives Considered

### Alternative 1: Keep interfaces + separate factories
**Pros:** Lighter weight, backward compatible
**Cons:** Factory functions separate from type, can drift
**Decision:** Rejected - classes provide better encapsulation

### Alternative 2: Specialized event handler base classes
**Pros:** Explicit CreateEvent vs UpdateEvent semantics
**Cons:** Two class hierarchies, more complexity, doesn't solve timestamp issue
**Decision:** Rejected - empty state pattern is simpler

### Alternative 3: Keep null checks, make them easier
**Pros:** Minimal changes
**Cons:** Doesn't eliminate boilerplate, type safety still limited
**Decision:** Rejected - doesn't solve the root problem

## Open Questions

### Q: Does `...state` spread work with class instances?
**A:** Yes! Spreading class instances works the same as objects in TypeScript. All properties are copied.

### Q: What about backward compatibility?
**A:** Breaking change. Will require major version bump (v0.2 → v0.3) and migration guide.

### Q: Performance impact of class vs interface?
**A:** Negligible. Classes are just objects at runtime. Empty state factory is called once per aggregate.

### Q: Can users still use interfaces?
**A:** No - `BaseState` must be a class for the `empty()` factory method to work.
